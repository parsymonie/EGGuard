# EGGuard

**Sync the [Université Toulouse Capitole (UT1)][ut1] blacklist categories and [abuse.ch][abusech] malware feeds into [EnforceGate vX][eg].**

EGGuard is a small, dependency-light Python tool that runs inside the
EnforceGate vX **toolbox sidecar**. On each run it downloads the domain feeds
you select, converts each one into an EnforceGate domain list plus a matching
`.policy` rule, and triggers a single live engine reload.

It turns the UT1 corpus — ~65 categories spanning adult content, malware,
phishing, gambling, social networks, VPN/proxy, redirectors and more — plus
abuse.ch's URLhaus and ThreatFox malware feeds into ready-to-enforce category
filtering, on-box, with no cloud lookups.

```
$ egguard update
INFO adult: updated (4,612,003 domains)
INFO malware: updated (259,965 domains)
INFO phishing: updated (260,104 domains)
INFO urlhaus: updated (570 domains)
INFO threatfox: updated (43,871 domains)
...
INFO loaded 5,196,517 domains across 65 rules
```

## How it works

```
  domain feeds (UT1 + abuse.ch)
        │   HTTPS — UT1 category tarballs, abuse.ch hostfiles
        ▼
  egguard update            (inside the toolbox sidecar)
   1. fetch each selected feed        (conditional GET, cached)
   2. extract & normalise domains
   3. write  <source>-<cat>.list      → /etc/enforcegate-shared/lists/
   4. write  60-<source>-<cat>.policy → /etc/enforcegate-shared/rules.d/
   5. engine.reload()                 (once, at the end)
        │
        ▼
  EnforceGate engine enforces
```

For each selected feed EGGuard writes two files into the shared volume that the
toolbox hands to the engine (`<source>` is `ut1` or `abusech`):

| File | Location | Purpose |
| ---- | -------- | ------- |
| `<source>-<category>.list` | `/etc/enforcegate-shared/lists/` | one domain per line (e.g. `ut1-adult.list`, `abusech-urlhaus.list`) |
| `60-<source>-<category>.policy` | `/etc/enforcegate-shared/rules.d/` | a single `match-domain-list` rule |

Generated policies are named `<source>-<category>` so they are clearly
attributed in the engine's audit log, and live at precedence `60-` by default —
after any operator-authored overrides. Because toolbox-generated rules can never
silently override hand-authored policy (the engine loads operator rules first),
your own allow/deny rules always win.

A generated policy looks like:

```
# Auto-generated by EGGuard — do not edit by hand.
# Source: https://dsi.ut-capitole.fr/blacklists/  (CC BY-SA 4.0)
# Regenerate with: egguard update
# Category: adult

ut1-adult: {
    match-domain-list: /etc/enforcegate-shared/lists/ut1-adult.list
    application: https
    action: deny
    description: Adult content — erotic to hard-core pornography
}
```

abuse.ch feeds produce the same shape under their own `abusech-` prefix, with
the source line pointing at the relevant abuse.ch project (e.g. URLhaus).

## Install

EGGuard targets the EnforceGate vX toolbox. Since 2026.32.0 the toolbox runs
Debian (bookworm-slim) and ships Python 3 with `python3-requests` and
`python3-yaml` already installed, so EGGuard runs there with **no `pip install`
step**. Deliver it through the toolbox's git-repo path:

```bash
# On the EnforceGate host
eghost toolbox enable                         # if not already on
eghost toolbox repo add egguard \
    https://github.com/parsymonie/egguard.git \
    --entrypoint scripts/egguard
```

Then schedule it (UT1 updates 2–3× per week, the abuse.ch feeds more often;
daily is a sensible default):

```bash
eghost toolbox cron edit
# add:
#   15 3 * * *  python3 /var/lib/enforcegate-toolbox/repos/egguard/scripts/egguard update
```

Install the categories you want once (this also populates their lists). Run it
inside the toolbox, where Python and the cloned repo live:

```bash
eghost toolbox shell
python3 /var/lib/enforcegate-toolbox/repos/egguard/scripts/egguard install adult malware phishing
```

### Local / development install

```bash
git clone https://github.com/parsymonie/egguard.git
cd egguard
pip install -e ".[dev]"      # package + dev tools (pytest, ruff, mypy)
egguard --help
```

To just run it from the clone (e.g. via `scripts/egguard`) without installing
the package, install only the runtime dependencies:

```bash
pip install -r requirements.txt
```

## Usage

```
egguard [--config PATH] [-v] <command>

Commands:
  install CATEGORY ...     download, write list/policy, reload (add categories)
  update [CATEGORY ...]    refresh installed categories (default: all installed)
  remove CATEGORY ...      delete a category's list/policy and reload
  select                   pick categories in a curses UI, then install/update
  list                     print the catalogue (source, installed marker, action)
  version                  print the version
  help                     show this help

install / update options:
  --action ACTION          set the action for these categories
                           (deny|warn|aup|permit; aliases: block=deny, allow=permit)
  -n, --dry-run            show what would change, but write/delete nothing
```

Running `egguard` with no command prints this help. It behaves like a package
manager: `install` adds categories, `update` refreshes the ones already
installed (this is the cron command), and `remove` drops them. An `--action`
given on `install` is remembered, so a later `update` keeps it; otherwise it
wins over config and catalogue suggestions for that run.

Prefer a menu? `egguard select` opens a curses picker: arrow keys to move,
`space` to toggle a category, `a` to cycle the action, `enter` to install the
selection (`q` cancels). Pre-checked rows are the already-installed categories.

Examples:

```bash
egguard install adult malware          # add two UT1 categories
egguard install urlhaus threatfox      # add abuse.ch feeds (needs an Auth-Key)
egguard install adult --action deny    # add adult, force (and remember) deny
egguard update                         # refresh everything installed (cron job)
egguard update adult                   # refresh just one
egguard remove gambling                # drop a category
egguard list                           # catalogue, with source and installed marked
```

### Exit codes

| Code | Meaning |
| ---- | ------- |
| `0` | all selected categories refreshed (or already current) |
| `1` | one or more categories failed; the rest succeeded |
| `2` | fatal — bad config, unknown category, or shared volume not mounted |

Each run also emits a one-line JSON summary suitable for SIEM ingestion via
`eghost toolbox logs`:

```json
{"domains":5196517,"dry_run":false,"elapsed_s":48.2,"event":"egguard-refresh","failed":0,"reloaded":true,"rules":65,"total":65,"updated":65}
```

## Use as a library

Every building block EGGuard uses is importable, so you can automate
list/policy creation for a category from your own Python. This is a complete,
runnable script — it downloads one category and writes its list and policy:

```python
from pathlib import Path

from egguard import Action, Fetcher, extract_domains, get, render_policy

category = get("gambling")                 # look up any category (UT1 or abuse.ch)
list_path = Path("ut1-gambling.list")
policy_path = Path("60-ut1-gambling.policy")

# 1. download the category tarball (conditional GET + retries built in).
fetcher = Fetcher(
    "https://dsi.ut-capitole.fr/blacklists/download",
    timeout=120,
    retries=3,
    user_agent="egguard-example/1.0",
)
result = fetcher.fetch(category)           # unconditional one-off fetch

# 2. normalise/de-duplicate the hosts and write the list.
domains = extract_domains(result.content, category.fmt)
list_path.write_text("\n".join(domains) + "\n")

# 3. render the matching .policy with the action you want.
policy_path.write_text(
    render_policy(category, list_path=list_path, action=Action.DENY)
)

print(f"wrote {len(domains)} domains to {list_path} and {policy_path}")
```

For an abuse.ch feed, pass `abusech_base_url` and `abusech_auth_key` to
`Fetcher` (the key is sent as the `Auth-Key` header). For the full
fetch→write→reload pipeline (with caching and the engine reload), use
`Refresher` with a `Config` and a bridge from `get_bridge()`.
`egguard.CATALOGUE` is the full list of `Category` records.

## Configuration

EGGuard reads `/var/lib/enforcegate-toolbox/config.yaml` (override with
`--config`). Every key is optional. See [`deploy/config.yaml`](deploy/config.yaml)
for a fully-commented template. The most useful knobs:

```yaml
# Two-digit precedence prefix for generated .policy files (after operator rules).
policy_prefix: "60"

# Default action for categories without an explicit override below.
default_action: deny

# Per-category action overrides: deny | warn | aup | permit
actions:
  child:           permit     # UT1 whitelist for young children
  liste_bu:        permit
  press:           warn
  social_networks: warn
  adult:           deny
  malware:         deny
  phishing:        deny

# Process only these categories (empty = all). Overrides `skip`.
include: []

# Skip these categories entirely.
skip:
  - examen_pix      # French PIX-exam list, irrelevant outside FR exams
  - tricheur_pix
```

If you set neither an override nor `default_action`, EGGuard falls back to a
**suggested disposition** baked into the catalogue (e.g. `adult`/`malware` →
`deny`, `child`/`liste_bu` → `permit`, `press`/`social_networks` → `warn`).
Run `egguard list` to see exactly what will be applied.

### Whitelists

UT1 ships some categories as *whitelists* rather than block lists — notably
`child`, `liste_bu`, and `examen_pix`. EGGuard maps these to `permit` by
default. Place them at a **lower** precedence than your deny rules (e.g. a
`config.yaml` that gives them their own prefix, or an operator rule at `10-`)
so the permit is evaluated first.

### abuse.ch feeds

Besides UT1, EGGuard can pull domain feeds from [abuse.ch][abusech]:

| Category | Source | Domains (approx.) | Default |
| -------- | ------ | ----------------- | ------- |
| `urlhaus` | URLhaus | ~600 active malware-distribution hosts | `deny` |
| `threatfox` | ThreatFox | ~44,000 malware-IOC domains, many families | `deny` |

(abuse.ch's other projects — Feodo Tracker, SSLBL, MalwareBazaar — publish IPs,
TLS fingerprints and file hashes rather than domains, so they don't fit
EGGuard's domain-list model.) The abuse.ch feeds need a **free Auth-Key** from
[auth.abuse.ch](https://auth.abuse.ch/); set it in the config and the feeds
become installable like any other category:

```yaml
abusech_auth_key: "your-abuse-ch-auth-key"
```

To keep the secret out of a file on disk, leave the config key unset and pass
it through the environment instead (the config value wins when both are set):

```bash
export EGGUARD_ABUSECH_AUTH_KEY="your-abuse-ch-auth-key"
egguard install urlhaus threatfox
```

The key is sent as the `Auth-Key` HTTP header, so it never appears in a URL.
Confirm your abuse.ch account's email after signing up — until you do, the
downloads return `403 Email address not confirmed`. Without a key the abuse.ch
feeds are skipped, and `install urlhaus` says so plainly. The key is held only
in your config or environment and is never written to logs or error messages.

## Air-gapped / mirrored deployments

For restricted networks, mirror the UT1 feed locally (rsync is supported
upstream) and point EGGuard at your mirror:

```yaml
base_url: "https://mirror.internal.example/ut1/download"
```

If your mirror uses a private CA, import it into the toolbox first
(`eghost toolbox ca import my-ca`) and attach it on the repo/fetch.

---

## What EGGuard does *not* do

- It does **not** redistribute UT1 or abuse.ch data — it downloads it at runtime.
- It does **not** mount or touch the engine's SSL-bump CA.
- It cannot override hand-authored operator policy; the engine's precedence
  rules guarantee operator rules win.
- Malformed lists it produces are rejected at the engine's next reload, leaving
  the previously-loaded ruleset live.
